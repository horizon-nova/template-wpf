## §1 總則

### §1.1 目的

本規範界定撰寫、修改、驗證程式碼時之行為準則。條文採「必須 / 禁止 / 反例 / 正例」格式,以便 AI 代理人與人類開發者均可逐條對照判定。

### §1.2 規範用語

- **必須 (MUST)**:強制條款,違反即視為不合規。
- **禁止 (MUST NOT)**:強制禁止條款。
- **應 (SHOULD)**:建議條款,偏離時須於程式碼評審中具備合理理由。

### §1.3 閱讀順序

進行任何修改前,必須依序:

1. 閱讀本規範全文
2. 閱讀 ARCH-001 適用章節
3. 閱讀欲修改功能之既有實作與其呼叫鏈
4. 閱讀 UI 與功能描述,確認需求之具體行為

## §2 語言與表述

### §2.1 文字語言

- 必須:所有註解、日誌、文件、文字使用**繁體中文**。
- 禁止:使用簡體中文。

### §2.2 狀態標記

- 必須:使用文字標記表示狀態,如 `[成功]`、`[失敗]`、`[警告]`、`[略過]`。
- 禁止:在註解、日誌、UI 文字中使用 emoji。

## §3 系統優先原則

### §3.0 解析思維

在進行任何設計或撰寫之前，無論規模大小、無論是新功能還是修改既有程式碼，必須先完成以下解析。這不是針對特定專案的規則，而是每一次動手之前都必須經歷的思考過程。

**必須在開始任何實作之前，能夠回答以下三個問題：**

1. 這件事的本質是什麼？它的輸入是什麼、最終結果只有哪幾種可能？
2. 在這個系統或環境中，答案會在什麼時機自然出現？
3. 基於以上兩點，我實際需要寫的是什麼？

這三個問題必須在動手之前回答，不是事後檢查。如果無法回答第一和第二個問題，代表解析尚未完成，不得開始實作。

**這三個問題不是步驟清單，是解析的起點。** 它們的目的不是讓你照順序執行，而是讓你在動手之前真正理解這件事的本質與機制。當你能夠回答這三個問題，你需要寫的程式碼會自然浮現，而且會比原本預期的少很多。

任何在解析完成之前就開始寫的程式碼，無論看起來多合理，都是在用程式碼猜一個系統已經知道的答案。這些程式碼不只多餘，它們會隨著業務成長不斷累積，最終讓程式碼變得混亂且失控。

### §3.1 原則陳述

系統、框架、語言已經提供之能力必須**直接使用**,禁止以手寫流程重新實作等效行為。撰寫新功能時,先確認平台是否已提供對應能力;若已提供,直接採用;若未提供,才考慮自行實作。

本原則之目的:讓程式碼依賴穩定之系統行為,而非依賴作者當下之條件判斷。當需求改變時,只需調整規則或參數,不需重寫流程。

### §3.2 具體條款

本節每一條均為可機械判定之行為規則。出現反例樣態即視為違反,必須依正例修正。

#### §3.2.1 檔案與資源存在性檢查

**禁止:**手寫存在性檢查後再呼叫 API。

**必須:**直接呼叫 API,由 API 之冪等性或例外機制處理。

**反例:**
```csharp
if (!Directory.Exists(path))
    Directory.CreateDirectory(path);
if (File.Exists(file))
    File.Delete(file);
File.WriteAllText(file, content);
```

**正例:**
```csharp
Directory.CreateDirectory(path);      // 對既存目錄為冪等操作
File.WriteAllText(file, content);     // 覆寫既存檔案
```

**理由:**`CreateDirectory` 對既存目錄不報錯;`WriteAllText` 覆寫既存檔案。手寫存在性檢查屬於重寫系統既有機制。

#### §3.2.2 空值處理

**禁止:**以 `if (x != null)` 鏈串處理可能為空之屬性路徑。

**必須:**使用語言提供之空值安全運算子(`?.`、`??`、`??=`)。

**反例:**
```csharp
string name = null;
if (user != null)
    if (user.Profile != null)
        if (user.Profile.DisplayName != null)
            name = user.Profile.DisplayName;
if (name == null)
    name = "匿名";
```

**正例:**
```csharp
var name = user?.Profile?.DisplayName ?? "匿名";
```

#### §3.2.3 資源釋放

**禁止:**手寫 `try-finally` 呼叫 `Close` / `Dispose`。

**必須:**使用 `using` 宣告或 `using` 區塊。

**反例:**
```csharp
var stream = File.OpenRead(path);
try
{
    stream.Read(buffer, 0, buffer.Length);
}
finally
{
    stream.Close();
    stream.Dispose();
}
```

**正例:**
```csharp
using var stream = File.OpenRead(path);
stream.Read(buffer, 0, buffer.Length);
```

#### §3.2.4 字串解析與比對

**禁止:**以字元逐位判斷、手寫狀態機處理可用 `Regex`、`Split`、`Parse` 完成之字串處理。

**必須:**使用語言與框架提供之解析 API。

**反例:**
```csharp
// 判斷是否為 4 碼數字 + 4 碼字母
bool valid = true;
for (int i = 0; i < 4; i++)
    if (!char.IsDigit(code[i])) { valid = false; break; }
for (int i = 4; i < 8; i++)
    if (!char.IsLetter(code[i])) { valid = false; break; }
```

**正例:**
```csharp
bool valid = Regex.IsMatch(code, @"^\d{4}[A-Za-z]{4}$");
```

#### §3.2.5 集合操作

**禁止:**以手寫迴圈實作可由 LINQ / 集合 API 完成之過濾、映射、聚合、查找。

**必須:**使用 `Where`、`Select`、`FirstOrDefault`、`Any`、`All`、`GroupBy` 等 API。

**反例:**
```csharp
var result = new List<string>();
foreach (var m in modules)
    if (m.IsEnabled && m.Version >= 2)
        result.Add(m.Name);
```

**正例:**
```csharp
var result = modules
    .Where(m => m.IsEnabled && m.Version >= 2)
    .Select(m => m.Name)
    .ToList();
```

#### §3.2.6 條件分派

**禁止:**針對「值 → 行為」之映射關係,以 `if-else if` 或 `switch` 長鏈實作。

**必須:**採用查表(`Dictionary`、規則表、策略表),由資料結構驅動行為。

**反例:**
```csharp
string GetLabel(ModuleStatus status)
{
    if (status == ModuleStatus.Active) return "使用中";
    else if (status == ModuleStatus.Disabled) return "已停用";
    else if (status == ModuleStatus.Pending) return "待驗證";
    else if (status == ModuleStatus.Error) return "錯誤";
    else return "未知";
}
```

**正例:**
```csharp
static readonly Dictionary<ModuleStatus, string> StatusLabels = new()
{
    [ModuleStatus.Active]   = "使用中",
    [ModuleStatus.Disabled] = "已停用",
    [ModuleStatus.Pending]  = "待驗證",
    [ModuleStatus.Error]    = "錯誤",
};

string GetLabel(ModuleStatus status)
    => StatusLabels.TryGetValue(status, out var label) ? label : "未知";
```

**判定式:**若條文會隨「狀態種類」「檔案類型」「指令名稱」等成員變多而成比例增加 `if` 分支,即屬此條禁止範圍。

#### §3.2.7 例外處理

**禁止:**以回傳值(bool、null、-1、字串代碼)取代例外機制,再於上層以 `if` 檢查。

**必須:**讓系統 API 之例外自然向上拋出,於需處理之層級集中捕捉。

**反例:**
```csharp
bool TryLoadModule(string path, out Module module)
{
    module = null;
    if (!File.Exists(path)) return false;
    try
    {
        module = LoadFrom(path);
        return true;
    }
    catch { return false; }
}
// 呼叫端
if (!TryLoadModule(p, out var m)) { /* 不知道失敗原因 */ }
```

**正例:**
```csharp
Module LoadModule(string path) => LoadFrom(path);  // 讓例外向上拋

// 呼叫端集中於業務邊界處理
try { var m = LoadModule(p); }
catch (FileNotFoundException) { /* 明確處理 */ }
catch (InvalidModuleException) { /* 明確處理 */ }
```

**例外:**語言或框架慣例明確要求 `Try...` 模式者(如 `int.TryParse`)不在此限。

### §3.3 修正原則

#### §3.3.1 原地修正

**禁止:**保留錯誤邏輯,並在其外層以新增 `if` 或新增包裝函式遮蔽錯誤。

**必須:**依下列順序修正:

1. 定位錯誤所在
2. 移除錯誤邏輯、錯誤分支、錯誤補丁
3. 回到正確結構
4. 重新實作

**反例:**
```csharp
// 原有 bug:路徑包含空白時會失敗
void SaveFile(string path) { /* ... */ }

// 錯誤修法:外層加判斷繞過問題
void SaveFileSafe(string path)
{
    if (path.Contains(" "))
        path = path.Replace(" ", "_");  // 改變了使用者輸入
    SaveFile(path);
}
```

**正例:**直接修正 `SaveFile` 內部對含空白路徑之處理邏輯。

#### §3.3.2 需求範圍

**禁止:**
- 超出需求擴展功能。
- 單一變更同時處理多個不相關議題。

**必須:**修改範圍與需求範圍一致。需求之外之改善建議另提 issue,不混入當次變更。

## §4 驗證規範

### §4.1 自我審核流程

完成任何修改後,必須依下列順序執行自我審核。每一步驟明確產出結果,不得略過。

**步驟 1:條款對照**

逐條對照附錄 A 之檢查清單,逐項回答「符合 / 不符合 / 不適用」。不得以「應該符合」「大致符合」回覆。

**步驟 2:反例掃描**

逐條對照 §3.2 之七項反例樣態,檢查當次變更是否出現:

- 手寫存在性檢查(§3.2.1)
- `if != null` 鏈串(§3.2.2)
- 手寫 `Close`/`Dispose`(§3.2.3)
- 手寫字串解析(§3.2.4)
- 手寫迴圈集合操作(§3.2.5)
- `if-else if` 長鏈分派(§3.2.6)
- 布林回傳遮蔽例外(§3.2.7)

出現任一樣態即為不合規,必須依正例修正。

**步驟 3:需求對照**

逐項回答:
- 本次修改解決了哪一條需求?(引用需求來源)
- 本次修改未涉及之需求是否被無意修改?
- 本次修改是否引入需求未要求之功能?

**步驟 4:ARCH-001 對照**

檢查本次修改是否觸及架構邊界:
- Views 是否直接操作 Utilities / 持久化?
- Services 是否出現業務判斷?
- Core 是否出現檔案模型或業務流程?
- 依賴方向是否違反 ARCH-001 §5?

### §4.2 測試驗證

- 必須:完成修改後進行測試驗證。
- 必須:驗證功能是否符合需求、是否產生例外、是否符合預期行為。

### §4.3 視覺驗證

涉及 UI、樣式、排版、畫面確認時:

- 必須:使用指定視窗截圖。
- 禁止:預設使用全螢幕截圖。
- 若指定視窗截圖失敗或邊界不正確,必須先修正截圖方式,再進行畫面確認。
- 僅當明確需要檢查整體桌面互動時,才可使用全螢幕截圖。

### §4.4 測試記錄

**必須:**
- 每次測試記錄測試項目與測試過程於文件中。
- 記錄測試方法(cmd、shell、Python 等)。
- 記錄測試結果(`[成功]` / `[失敗]`)。
- 失敗時記錄失敗原因與所用測試方式。
- 下次測試前先查閱既有記錄。
- 若某測試方式已標記失敗,下次優先嘗試其他方式。

**記錄位置:**`testing-guidelines.md` 或專案指定之測試項目文件。

### §4.5 自我修復

本條款規範 AI 代理人於驗證過程中之自主修復行為。

#### §4.5.1 適用條件

**必須全部滿足方可自我修復:**
- 測試結果明確(非「可能失敗」「似乎有問題」)。
- 問題原因清楚,可具體指向程式碼位置。
- 修復前已查閱既有測試記錄。
- 修復方案不違反本規範或 ARCH-001 任何條款。

**修復後必須:**
- 再次進行測試驗證。
- 更新測試記錄,載明修復方法與結果。

#### §4.5.2 停止條件

出現下列任一情況,必須停止自我修復並回報開發者:

- 自我修復後問題仍存在。
- 問題原因不清楚或不確定。
- 修復可能產生其他副作用。
- 同一測試方式已失敗兩次以上。
- 修復方案需違反本規範任一條款。

#### §4.5.3 絕對禁止

- 禁止:重複使用已標記為失敗之測試方式。
- 禁止:問題原因未釐清時自行修復。
- 禁止:以新增 `if` 分支或新增包裝函式遮蔽錯誤(違反 §3.3.1)。
- 禁止:在修復過程中擴展至需求範圍外之其他修改。

## §5 違規處理

### §5.1 發現違規時

**必須:**
- 立即停止所有修改。
- 說明違規內容、所屬條款、違反原因。
- 提出修正方案(僅說明,不執行)。
- 等待開發者明確確認後方可繼續。

### §5.2 禁止自行修正之情境

- 除開發者明確要求外,禁止自行修正違規。
- 禁止在未釐清違規內容與原因時修改。
- 禁止在下次對話中假設開發者已確認而直接修改。

### §5.3 回報格式

回報違規時使用下列格式:

```
[違規項目] <條款編號,如 §3.2.6>
[違規位置] <檔案:行號>
[違規內容] <具體描述>
[違反原因] <為何構成違規>
[修正方案] <建議如何修正,不執行>
```

## 附錄 A 自我審核檢查清單

## 附錄 B 反例對照速查

| 樣態 | 反例特徵 | 正解 | 條款 |
| --- | --- | --- | --- |
| 存在性檢查 | `if (!File.Exists) / if (File.Exists)` 後呼叫同族 API | 直接呼叫,由 API 或例外處理 | §3.2.1 |
| 空值鏈串 | 巢狀 `if (x != null)` | `?.` / `??` / `??=` | §3.2.2 |
| 手動釋放 | `try-finally` 呼叫 `Close`/`Dispose` | `using` 宣告或區塊 | §3.2.3 |
| 手寫解析 | 字元逐位判斷、手寫狀態機 | `Regex` / `Split` / `Parse` | §3.2.4 |
| 迴圈過濾 | `foreach` + `if` 收集結果 | `Where` / `Select` / `Any` | §3.2.5 |
| 值分派 | `if-else if` 長鏈 / `switch` 僅回傳值 | `Dictionary` 查表 | §3.2.6 |
| 布林遮蔽 | `bool TryXxx` 隱藏例外原因 | 拋出具體例外,於邊界捕捉 | §3.2.7 |
| 外層包裝 | 錯誤邏輯保留,外層加 `if` 繞過 | 移除錯誤邏輯,原地修正 | §3.3.1 |
